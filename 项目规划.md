# CF-Workers-SUB-Next 项目规划

> 新一代 Cloudflare Workers 订阅汇聚与规则生成工具

---

## 一、现状分析

### 现有项目（CF-Workers-SUB）

| 维度 | 现状 |
|------|------|
| 部署方式 | Cloudflare Workers / Pages + KV |
| 认证方式 | Token 访问（无 UI 界面） |
| 订阅管理 | 通过编辑环境变量或 KV 编辑页面添加链接 |
| 规则格式 | `.ini` 外部配置（subconverter 格式） |
| 转换后端 | 依赖第三方 SUBAPI（sub.cmliussss.net） |
| 规则来源 | ACL4SSR 的 `.list` / `.ini` 规则 |
| 作者维护 | 上游已停更 |

### 核心痛点

1. **.ini 规则格式已过时** — Clash Meta / Mihomo 原生支持的是 `rule-providers`（YAML 格式），.ini 是 subconverter 时代的产物，需要额外转换层
2. **无管理界面** — 每次修改订阅链接需要进 KV 或环境变量，操作门槛高
3. **依赖第三方转换后端** — 节点信息经第三方服务器，存在隐私风险
4. **规则不可选** — 固定一套 ACL4SSR 多国规则，无法按需切换

## 二、现代规则格式对比

### 旧格式（.ini - subconverter 外部配置）

```ini
[custom]
ruleset=🚀 节点选择,https://.../ProxyLite.list
ruleset=🎯 全球直连,https://.../LocalAreaNetwork.list
ruleset=🛑 全球拦截,https://.../BanAD.list
```

- 需要 subconverter 后端解析
- 规则文件是 `.list` 纯文本格式
- 无法直接用于 Clash Meta / Mihomo

### 新格式（YAML - rule-provider）

```yaml
# rule-provider 定义
rule-providers:
  ProxyLite:
    type: http
    behavior: classical
    url: "https://raw.githubusercontent.com/ACL4SSR/ACL4SSR/master/Clash/Providers/ProxyLite.yaml"
    interval: 86400
    path: ./rules/ProxyLite.yaml

# 规则引用
rules:
  - RULE-SET,ProxyLite,🚀 节点选择
  - RULE-SET,BanAD,🛑 全球拦截
  - GEOIP,CN,🎯 全球直连
  - MATCH,🐟 漏网之鱼
```

- Clash Meta / Mihomo / OpenClash 原生支持
- 支持自动更新规则（interval 参数）
- YAML 格式更规范，可读性更强
- ACL4SSR 已经提供了 YAML 版本（`/Clash/Providers/` 目录）

## 八、Cloudflare Workers 技术可行性

| 要求 | 可行性 | 说明 |
|------|--------|------|
| 密码登录 | ✅ 简单 | Workers 支持 Session/Cookie，密码用环境变量配置 |
| Web UI | ✅ 可行 | 单页 HTML 内嵌在 Worker 返回中，无需额外部署 |
| KV 存储 | ✅ 可行 | 免费 1GB 存储，1000 次写入/天，足够个人使用 |
| 订阅抓取 | ✅ 可行 | Workers 支持 fetch API 抓取外部订阅 |
| YAML 生成 | ✅ 可行 | 纯字符串拼接，Workers 原生支持 |
| 规则引用 | ✅ 可行 | 直接输出 rule-provider 的 HTTP URL 引用 |
| 节点解析 | ✅ 已实现 | 支持 vmess/vless/trojan/ss/ssr/hysteria2/tuic 协议解析 |
| 节点健康检测 | ✅ 简化版 | 简化版健康检测（去重+统计），TCP 检测可增强 |

**关键限制：**
- Workers 免费计划每天 10 万次请求，CPU 时间 10ms/请求
- 节点解析需要自己实现协议解析器（已实现 vmess/vless/trojan/ss/ssr/hysteria2/tuic）
- 无法直接 TCP 测速（Workers 限制），简化为节点统计
- KV 写入有配额限制（每天 1000 次）

## 十一、安全架构

### 秘密管理策略

所有敏感信息通过 **GitHub Repository Secrets** 管理，`wrangler.toml` 不含任何敏感信息：

| 信息 | 存储位置 | 注入方式 | 说明 |
|------|---------|---------|------|
| `CLOUDFLARE_API_TOKEN` | GitHub Repository Secrets | CI 中 `wrangler deploy` 通过 `env` 传入 | 用于认证 CF 部署 |
| `KV_NAMESPACE_ID` | GitHub Repository Secrets | CI 中 `sed` 写入 `wrangler.toml` | KV 命名空间标识 |
| `SUB_TOKEN` | GitHub Repository Secrets | `wrangler secret put` 部署为 Worker Secret | 订阅地址访问令牌 |

### 部署流程

```
git push → GitHub Actions 触发
  │
  ├─ 1. npm ci + npm run build（构建前端）
  ├─ 2. sed 写入 KV_NAMESPACE_ID 到 wrangler.toml
  ├─ 3. wrangler deploy（部署 Worker）
  └─ 4. wrangler secret put SUB_TOKEN（如设置）
       │
       ▼
  CF Workers 上线
       │
       ▼
  Worker 运行时：env.SUB_TOKEN 读取 Worker Secret
       │
       ▼
  /sub?token=xxx&user=xxx 校验
```

> Fork 用户只需配置自己的 GitHub Secrets 即可使用，无需修改任何代码。

## 十三、部署指南（最新版）

### 第一步：创建 KV 命名空间

1. 打开 [Cloudflare Dashboard](https://dash.cloudflare.com/)
2. Workers & Pages → KV → 创建命名空间
3. 名称：`SUB_NEXT_DATA`
4. 创建后复制 **命名空间 ID**

### 第二步：创建 Cloudflare API Token

1. [Cloudflare API Tokens](https://dash.cloudflare.com/profile/api-tokens) → **Create Token**
2. 选择 **Edit Cloudflare Workers** 模板
3. 创建后 **立即复制 Token**

### 第三步：Fork 并配置仓库

1. Fork 本仓库到你的 GitHub 账号
2. 进入你的仓库 → **Settings → Secrets and variables → Actions → Repository secrets**（不是 Environment secrets）
3. 添加以下三个 Secrets：

| Secret 名称 | 值 | 说明 |
|------------|-----|------|
| `CLOUDFLARE_API_TOKEN` | 刚复制的 API Token | 必填，用于部署 Worker |
| `KV_NAMESPACE_ID` | 第一步复制的 KV 命名空间 ID | 必填，CI 自动写入 wrangler.toml |
| `SUB_TOKEN` | 自定义字符串 | 可选，设置后订阅地址需带此 token 才能访问 |

### 第四步：推送触发部署

```bash
git add .
git commit -m "配置项目"
git push origin main
```

GitHub Actions 会自动：
1. 将 `KV_NAMESPACE_ID` 写入 `wrangler.toml`
2. 用 `wrangler deploy` 部署 Worker
3. 用 `wrangler secret put` 注入 `SUB_TOKEN`（如设置了）

### 第五步：访问和配置

1. 打开 Cloudflare Dashboard → Workers & Pages → `sub-next` → 访问分配域名
2. 用密码 `admin` 登录
3. **订阅管理** → 添加你的订阅链接/节点链接
4. **规则管理** → 勾选需要的规则集
5. **输出配置** → 复制对应格式的订阅地址到 OpenClash
6. **管理后台** → 修改密码、设置链接替换（GitHub 镜像加速）

### 可选：配置 SUB_TOKEN

部署后如果不想公开订阅地址，在 GitHub 仓库中设置 `SUB_TOKEN` Secret：
1. 仓库 **Settings → Secrets and variables → Actions → Repository secrets**
2. 添加 `SUB_TOKEN`，值填你想要的自定义 Token
3. 重新推送代码或手动触发 Actions 部署

部署完成后，订阅地址需要带 `?token=你的自定义Token` 参数才能访问。

**安全说明：**
- `CLOUDFLARE_API_TOKEN` 和 `SUB_TOKEN` 都存储在 GitHub Secrets 中，不会暴露在代码里
- `wrangler.toml` 中的 KV 命名空间 ID 是 CF 资源标识符，不是敏感信息，可放心保留
- 订阅链接支持 `user` 参数指定用户：`/sub?token=xxx&user=用户名`

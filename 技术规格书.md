## 二、Cloudflare KV 数据结构

KV 是整个项目的持久化存储，遵循最小化原则，只存必要数据。

### 2.1 KV 命名空间

| 命名空间 | 用途 | 备注 |
|---------|------|------|
| `SUB_NEXT_DATA` | 存储所有用户数据 | Workers 绑定名 |

### 2.2 Key 设计

| Key | 存储内容 | 格式 | 说明 |
|-----|---------|------|------|
| `admin:hash` | 管理密码的哈希 | `{hash: "<sha256>", salt: "<uuid>"}` | 登录验证 |
| `sessions:<token>` | 登录会话 | `{username: "admin", expires: 1234567890}` | token 随机生成，有过期时间 |
| `subscriptions` | 订阅链接列表 | `["url1", "url2", ...]` | 纯数组，每个元素一个订阅 URL |
| `rules:enabled` | 已启用的规则集 ID 列表 | `["BanAD", "Netflix", "Google", ...]` | 仅存 ID，不存完整数据 |
| `link_replace:from` | 规则链接替换 - 原始前缀 | 字符串 | 如 `https://raw.githubusercontent.com/` |
| `link_replace:to` | 规则链接替换 - 替换后前缀 | 字符串 | 如 `https://ghproxy.net/...` |
| `users` | 用户列表 | `[{id, username, role, password_hash, salt, created_at, last_login}]` | 含管理员和普通用户 |
| `version` | 当前部署版本号 | `"v0.1.0"` | 用于升级检测对比 |

### 2.3 KV 读写策略

| 操作 | KV 操作 | 频率 | 说明 |
|------|---------|------|------|
| 登录验证 | GET `admin:hash` | 每次登录 | 读密集型 |
| 创建会话 | PUT `sessions:<token>` | 每次登录 | 7 天过期 |
| 获取规则状态 | GET `rules:enabled` | 每次页面加载 | 缓存可考虑 |
| 保存规则状态 | PUT `rules:enabled` | 点击保存时 | 低频 |
| 订阅列表 | GET/PUT `subscriptions` | 增删改时 | 低频 |
| 链接替换配置 | GET/PUT `link_replace:*` | 管理后台设置 | 极少变动 |

**优化：** 前端加载时可一次批量读取所有配置（用一个 KV key 存完整配置快照），减少 KV 读取次数。

## 三、API 接口设计

### 3.1 接口总览

所有 API 响应格式统一为：
```json
{ "ok": true, "data": {...} }
// 或
{ "ok": false, "error": "错误信息" }
```

#### 认证相关

| 方法 | 路径 | 说明 | 认证 |
|------|------|------|------|
| POST | `/api/login` | 密码登录 | ❌ |
| POST | `/api/logout` | 登出 | ✅ Token |
| GET  | `/api/session` | 检查当前会话是否有效 | ✅ Token |

#### 规则管理

| 方法 | 路径 | 说明 | 认证 |
|------|------|------|------|
| GET  | `/api/rules/list` | 获取全部规则集清单 | ✅ Token |
| GET  | `/api/rules/enabled` | 获取当前启用规则集 ID 列表 | ✅ Token |
| PUT  | `/api/rules/enabled` | 保存当前启用规则集 ID 列表 | ✅ Token |

#### 订阅管理

| 方法 | 路径 | 说明 | 认证 |
|------|------|------|------|
| GET  | `/api/subscriptions` | 获取订阅链接列表 | ✅ Token |
| POST | `/api/subscriptions` | 添加订阅链接 | ✅ Token |
| DELETE | `/api/subscriptions` | 删除订阅链接 | ✅ Token |

#### 配置输出

| 方法 | 路径 | 说明 | 认证 |
|------|------|------|------|
| GET  | `/api/config/preview` | 预览生成的 YAML 配置 | ✅ Token |
| GET  | `/sub?token=xxx` | 自动适配（返回 Clash YAML 格式） | ❌ 可选 |
| GET  | `/sub?token=xxx&format=clash` | Clash Meta / Mihomo 格式 | ❌ 可选 |
| GET  | `/sub?token=xxx&format=base64` | Base64 编码的节点列表 | ❌ 可选 |
| GET  | `/sub?token=xxx&format=singbox` | Sing-box 格式 | ❌ 可选 |
| GET  | `/sub?token=xxx&format=surge` | Surge 格式 | ❌ 可选 |
| GET  | `/sub?token=xxx&format=loon` | Loon 格式 | ❌ 可选 |

**关于订阅地址：**

- 访问 `/sub` 不带 token 也可使用（但建议加上 token 防止被他人滥用）
- 不指定 `format` 参数时，自动检测请求的 User-Agent 返回合适格式
- 每种格式生成的订阅文件都包含：**所有节点信息 + 对应格式的规则配置**

| POST | `/api/nodes/health-check` | 手动触发节点健康检测 | ✅ 管理员 |
| GET  | `/api/nodes/health-check` | 获取最近一次健康检测结果 | ✅ Token |

#### 管理后台

| 方法 | 路径 | 说明 | 认证 |
|------|------|------|------|

| POST | `/api/admin/users` | 添加用户 | ✅ 管理员 |
| DELETE | `/api/admin/users` | 删除用户 | ✅ 管理员 |
| PUT  | `/api/admin/password` | 修改管理密码 | ✅ 管理员 |
| GET  | `/api/admin/link-replace` | 获取链接替换配置 | ✅ 管理员 |
| PUT  | `/api/admin/link-replace` | 保存链接替换配置 | ✅ 管理员 |

#### 系统

| 方法 | 路径 | 说明 | 认证 |
|------|------|------|------|
| GET  | `/api/version` | 获取当前版本（用于升级检测） | ❌ |

### 3.2 登录认证流程

```
登录页面 → 输入密码
          ↓
     POST /api/login { password: "xxx" }
          ↓
  验证通过 → 生成随机 Session Token → KV 存储 → 返回 Set-Cookie
          ↓
  前端带 Cookie 访问任何 API → Worker 校验 Session → 放行或 401
```

- Session Token 使用 `crypto.randomUUID()` 生成
- Cookie 设置 `HttpOnly; Secure; SameSite=Lax; Max-Age=604800`（7 天）
- 普通用户和管理员共用同一套 session 机制，但角色权限在路由层区分

## 四、前端页面路由

单页应用（SPA），所有页面都在 `index.html` 中，由前端 JS 控制显示隐藏。

| 路由（Hash） | 对应页面 | 说明 |
|-------------|---------|------|
| `#/login` | 登录页 | 默认首页，未登录时显示 |
| `#/rules` | 规则管理 | 默认主页面，显示规则集列表 |
| `#/subscribe` | 订阅管理 | 管理订阅链接 |
| `#/output` | 输出配置 | 预览 YAML、复制订阅地址 |
| `#/help` | 使用说明 | 静态内容 |
| `#/admin` | 管理后台（模态框） | 用户管理、链接替换、系统设置 |

**页面加载流程：**

```
访问任意路径
    ↓
Worker 返回 index.html（SSR 或静态返回）
    ↓
前端检查 Cookie 中是否有 Session
    ↓
   ├─ 有 → GET /api/session 验证有效 → 进入主界面
   └─ 无 → 显示登录页
    ↓
（验证或登录后，根据 Hash 路由显示对应页面）
```

## 五、YAML 配置生成逻辑

### 5.1 核心生成流程

Worker 接收订阅地址或单节点链接后，执行以下流程：

```
用户输入的订阅链接 / 单节点链接
        ↓
Worker 发起 fetch 抓取所有订阅
        ↓
解析节点 → 提取所有协议节点（vmess/vless/trojan/ss/ssr/hysteria2）
        ↓
节点去重（按 server:port 去重）
        ↓
节点健康检测（TCP ping 测试可用性）
        ↓
移除无效节点
        ↓
按请求格式输出：
  ├─ Clash YAML  → 完整配置（节点 + 规则 + 策略组）
  ├─ Base64      → 纯节点列表 Base64 编码
  ├─ Sing-box    → Sing-box JSON 格式
  ├─ Surge       → Surge 格式
  ├─ Loon        → Loon 格式
  └─ 自适应      → 根据 User-Agent 自动选择
```

### 5.2 Clash YAML 输出结构

生成的是完整 Clash Meta / Mihomo 配置，包含：

1. **proxy-providers** — 从用户订阅链接生成
2. **rule-providers** — 从用户勾选的规则集生成（内置规则直接硬编码 GLOBAL 不走 providers）
3. **proxy-groups** — 策略组（节点选择、自动选择、直连、拒绝、漏网之鱼）
4. **rules** — 规则引用顺序

### 5.3 生成流程

```
用户勾选规则集 → 点击"保存配置"
        ↓
PUT /api/rules/enabled { ids: ["BanAD", "Netflix", "YouTube", ...] }
        ↓
写入 KV → 返回成功
        ↓
OpenClash 请求 /sub?token=xxx
        ↓
Worker 读取 KV → 获取启用规则集 + 订阅链接
        ↓
   ┌─────────────────────────────────────┐
   │ 1. 内置安全规则（始终加入）          │
   │    - GEOIP,CN → 直连               │
   │    - 国内直连 domain → 直连          │
   │    - 内网地址 ipcidr → 直连          │
   │    - 国内 IP 段 ipcidr → 直连        │
   ├─────────────────────────────────────┤
   │ 2. 用户勾选的 rule-providers         │
   │    - 如有链接替换，替换 URL 前缀      │
   ├─────────────────────────────────────┤
   │ 3. proxy-providers（从订阅链接生成）  │
   ├─────────────────────────────────────┤
   │ 4. proxy-groups（策略组）            │
   ├─────────────────────────────────────┤
   │ 5. rules（按优先级排序）             │
   └─────────────────────────────────────┘
        ↓
返回完整 YAML → OpenClash 自动加载
```

### 5.4 规则优先级

```
1. 🛑 全球拦截（REJECT）      — 广告/恶意域名
2. 🚀 代理规则               — ProxyLite / 用户勾选的代理规则
3. 🎯 国内直连               — 国内域名 / 国内 IP / 内网 IP
4. 🐟 漏网之鱼               — 未匹配流量（走节点选择）
```

### 5.5 链接替换逻辑

如果管理后台设置了链接替换：

```
原始前缀: https://raw.githubusercontent.com/
替换为:   https://ghproxy.net/https://raw.githubusercontent.com/

生成 rule-provider 时:
  url: "https://ghproxy.net/https://raw.githubusercontent.com/ACL4SSR/ACL4SSR/master/Clash/Providers/BanAD.yaml"
```

用于国内网络访问 GitHub raw 慢的场景。

## 六、规则集清单管理

### 6.1 清单结构

规则集清单是静态数据，硬编码在 `src/rulesets.js` 中，格式：

```javascript
const RULESETS = [
  {
    id: "BanAD",                     // 唯一 ID（对应 YAML 文件名，不含扩展名）
    name: "全球拦截",                 // 显示名称
    desc: "拦截常见广告、跟踪器、挖矿域名", // 说明
    group: "ad",                     // 分组（ad, media, google, ai, game, system, social, dev, finance, domestic, other）
    url: "https://raw.githubusercontent.com/ACL4SSR/ACL4SSR/master/Clash/Providers/BanAD.yaml",
    builtin: false,                  // 是否内置安全规则（始终启用）
    common: true                     // 是否常用（默认显示，还是点击"显示全部"才出现）
  },
  // ... 147 个条目
];
```

### 6.2 更新方式（方案二：人工更新）

当 ACL4SSR 新增了 YAML 规则文件：

1. 开发者修改 `src/rulesets.js`，新增条目
2. 更新 `version` 为新的版本号（如 `v0.1.1`）
3. `git push` → GitHub Actions 自动部署
4. 用户下次访问时，升级检测提示"发现新版本"
5. 用户刷新页面即看到新规则集

## 七、GitHub → Cloudflare Workers 自动部署

### 7.1 wrangler.toml 配置

```toml
name = "sub-next"
main = "src/index.js"
compatibility_date = "2024-12-01"

# KV 绑定
[[kv_namespaces]]
binding = "SUB_NEXT_DATA"
id = "xxxxxxxx"  # 用户需要手动在 CF Dashboard 创建 KV 并填入

# 环境变量
[vars]
SESSION_EXPIRY_HOURS = "168"  # session 过期时间（小时）
SUB_TOKEN = ""                # 订阅访问令牌（留空则无需令牌即可访问）
```

### 7.2 GitHub Secrets（需要用户在仓库中设置）

| Secret 名称 | 说明 | 获取方式 |
|------------|------|---------|
| `CLOUDFLARE_API_TOKEN` | Cloudflare API Token | CF Dashboard → My Profile → API Tokens → 创建 |

**API Token 权限要求：**
- `Workers R2` — Edit（读写 Workers）
- `Workers KV` — Edit（读写 KV）
- `Account.Account Analytics` — Read（可选，用于监控）

> ⚠️ **Token 保密措施：**
> - 在 GitHub 仓库 Settings → Secrets and variables → Actions 中设置，**永远不会出现在代码或日志中**
> - 本地开发时使用 `.env` 文件，已加入 `.gitignore`
> - 具体设置步骤见 `README.md` 的部署指南

### 7.3 GitHub Actions Workflow

```yaml
# .github/workflows/deploy.yml
name: Deploy to Cloudflare Workers

on:
  push:
    branches: [main]
  workflow_dispatch:  # 支持手动触发

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - name: Deploy to Cloudflare Workers
        uses: cloudflare/wrangler-action@v3
        with:
          apiToken: ${{ secrets.CLOUDFLARE_API_TOKEN }}
```

## 八、升级检测机制

### 8.1 检测原理

```
前端页面加载
    ↓
GET /api/version  →  获取当前 Worker 的版本号
    ↓
对比 GitHub Releases 最新版本
  fetch("https://api.github.com/repos/BobVane/CF-Workers-SUB-Next/releases/latest")
    ↓
  ├─ 一致 → ✅ 已是最新版本
  └─ 不一致 → 🆕 发现新版本（显示 Release 链接）
```

### 8.2 Fork 用户适配

如果别人 fork 了项目：

1. 在 `README.md` 中写明：fork 后请在 `src/index.js` 顶部修改 `GITHUB_REPO` 变量为你的仓库地址
2. 升级检测会自动对比你的 GitHub Releases，无论上游是否更新
3. 如果想跟踪上游更新，可以手动 Watch 原仓库

以下内容将在 `README.md` 中详细说明：

```
## 部署指南

### 前提条件
1. 一个 Cloudflare 账号（免费即可）
2. 一个 GitHub 账号

### 第一步：创建 KV 命名空间
1. 登录 CF Dashboard → Workers & Pages → KV
2. 创建命名空间，命名为 `SUB_NEXT_DATA`
3. 记下命名空间 ID

### 第二步：创建 API Token
1. CF Dashboard → My Profile → API Tokens → Create Token
2. 选择 "Edit Cloudflare Workers" 模板
3. 创建后复制 Token（离开页面后不再显示）

### 第三步：Fork 仓库并配置 Secrets
1. Fork 本仓库到你的 GitHub
2. 进入仓库 Settings → Secrets and variables → Actions → Repository secrets
3. 添加以下三个 Secrets：

| Secret 名称 | 值 | 说明 |
|------------|-----|------|
| `CLOUDFLARE_API_TOKEN` | 上一步复制的 API Token | 必填 |
| `KV_NAMESPACE_ID` | 第一步复制的 KV 命名空间 ID | 必填 |
| `SUB_TOKEN` | 自定义字符串 | 可选，设置后订阅地址需带此 token 才能访问 |

### 第四步：推送触发部署
1. `git add . && git commit -m "配置项目" && git push origin main`
2. GitHub Actions 自动执行：
   - `sed` 写入 `KV_NAMESPACE_ID` 到 `wrangler.toml`
   - `wrangler deploy` 部署 Worker
   - `wrangler secret put SUB_TOKEN` 注入订阅 Token（如设置了）

### 安全架构
```
git push
  ↓
GitHub Actions 触发
  ↓
1. 构建前端 (npm ci + npm run build)
2. 从 Secrets 读取 KV_NAMESPACE_ID → 写入 wrangler.toml
3. wrangler deploy（CLOUDFLARE_API_TOKEN 通过 env 传入）
4. wrangler secret put SUB_TOKEN（如设置了）
  ↓
CF Workers 上线
  ↓
Worker 运行时：env.SUB_TOKEN 读取 Worker Secret
                   ↓
                  /sub?token=xxx 校验
```

> 🔒 安全说明：
> - `wrangler.toml` 中不包含任何敏感信息
> - 所有敏感配置（API Token、KV ID、订阅 Token）均通过 GitHub Secrets 注入
> - 仓库可放心公开，便于他人 Fork 使用

## 十、设计决策总结

以下是对 UI 原型讨论中所有设计决策的结论记录：

| 序号 | 决策项 | 结论 |
|------|--------|------|
| 1 | 订阅地址 Token 管理 | 通过 GitHub Secrets 注入，`wrangler secret put` 部署为 Worker Secret。不存入仓库。留空则无需 token 即可访问 |
| 2 | 多订阅源 | 多个订阅 URL 合并节点，聚合为一个订阅输出 |
| 3 | 节点解析 | Worker 端抓取并解析所有协议节点（vmess/vless/trojan/ss/ssr/hysteria2），生成包含完整节点信息的订阅文件 |
| 4 | 输出格式 | 支持 6 种格式：自适应、Base64、Clash、Sing-box、Surge、Loon |
| 5 | 节点健康检测 | 自动检测无效节点和重复节点，去重并移除失效节点 |
| 6 | 默认规则模式 | 轻量模式（仅基础代理 + 国内直连 + 内网地址） |
| 6 | 初始密码 | `admin`，登录后可在后台修改 |
| 7 | 秘密管理 | KV 命名空间 ID 和 API Token 通过 GitHub Secrets 注入，SUB_TOKEN 通过 `wrangler secret put` 部署。`wrangler.toml` 不含任何敏感信息 |
| 8 | 代码部署 | 本地写代码，审核后推 GitHub，GitHub Actions 自动部署到 CF Workers |
| 8 | 安全规则锁定 | 国内直连、国内 IP 段、内网地址始终启用，不可关闭 |
